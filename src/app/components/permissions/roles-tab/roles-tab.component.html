How permissions work in Polari

Polari's permission model is layered on top of Keycloak's group → role → policy chain. You don't grant permissions to users directly — you grant them to roles, roles get attached to groups, and Keycloak's policies decide who is allowed to add users to which groups.

  • Roles (this tab) — define the CRUDE grants each role hands its holder, per object class. Editing here updates the matrix Polari uses to gate every backend call.
  • Groups (configured in Keycloak) — collections of users. A group's realmRoles list is the auto-grant: drop a user into a group and they inherit those roles in their token immediately.
  • Permissions / Policies (Keycloak's fine-grained admin authz) — let you grant "can manage group X" to an access-management role without giving that person the keys to modify code or roles themselves. Critical for onboarding in secure environments: intermediate admins add people to groups, principals stay locked.
